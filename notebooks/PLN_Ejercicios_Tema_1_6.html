Aplicación 1: Diagnóstico Clínico Asistido por PLN en Historiales Médicos¶

¿En qué consiste?
Sistemas como Amazon Comprehend Medical o Google Cloud Healthcare NLP API analizan automáticamente historiales clínicos en texto libre (notas de médicos, informes de alta, resultados de laboratorio) para extraer entidades médicas: diagnósticos, medicamentos, dosis, síntomas y relaciones entre ellos.

¿Por qué es útil e interesante?
Los médicos documentan en lenguaje natural, no en bases de datos estructuradas. El PLN permite convertir ese texto no estructurado en información accionable: detectar interacciones medicamentosas, identificar pacientes de riesgo, acelerar la codificación ICD-10 para facturación, y apoyar la investigación epidemiológica. Es una aplicación donde el PLN salva vidas literalmente, al reducir errores médicos por información dispersa o mal codificada.

Aplicación 2: Moderación Automática de Contenido en Redes Sociales¶

¿En qué consiste?
Plataformas como Meta, X (Twitter) y YouTube utilizan modelos de PLN para detectar y filtrar automáticamente contenido dañino: discurso de odio, desinformación, acoso, contenido violento o spam. Estos sistemas clasifican millones de publicaciones por segundo en múltiples idiomas.

¿Por qué es útil e interesante?
La escala del problema hace imposible la moderación humana exclusiva: Meta procesa más de 100 millones de publicaciones diarias. El PLN permite una respuesta en tiempo real, consistente y escalable. Además, es un problema técnicamente desafiante porque el lenguaje ofensivo evoluciona constantemente (nuevos eufemismos, ironía, contexto cultural), lo que exige modelos que aprendan continuamente. Tiene implicaciones éticas y sociales enormes.

Aplicación 3: Generación Automática de Código a partir de Lenguaje Natural¶

¿En qué consiste?
Herramientas como GitHub Copilot (basado en OpenAI Codex), Amazon CodeWhisperer o Google Gemini Code permiten a los desarrolladores describir en lenguaje natural lo que quieren hacer ('crea una función que ordene una lista de diccionarios por el campo fecha') y el modelo genera el código correspondiente en Python, JavaScript, Java u otros lenguajes.

Profundización en un Caso: Generación Automática de Código¶

Caso elegido: GitHub Copilot — desarrollado por GitHub en colaboración con OpenAI.

Descripción del caso concreto¶

GitHub Copilot fue lanzado en 2021 y se convirtió en el asistente de código con mayor adopción en la historia del desarrollo de software. En 2023, GitHub reportó que más de 1 millón de desarrolladores lo usaban activamente, y que el 46% del código en los proyectos donde estaba habilitado era generado por la IA.

¿Cómo funciona la parte de PLN?¶

GitHub Copilot se basa en OpenAI Codex, un modelo de la familia GPT entrenado específicamente sobre código fuente. La arquitectura es un Transformer de tipo decoder-only (similar a GPT-3/4), con las siguientes características clave:

  1. Preentrenamiento masivo: El modelo fue entrenado sobre decenas de millones de repositorios públicos de GitHub, aprendiendo patrones de código en más de 12 lenguajes de programación. Esto le da una comprensión profunda de la sintaxis, semántica y convenciones de cada lenguaje.

  2. Comprensión del contexto: El modelo recibe como entrada el código ya escrito en el archivo (contexto), los comentarios en lenguaje natural del desarrollador, y el nombre de la función o variable que se está escribiendo. Con esa información, predice los tokens más probables que completan el código.

  3. Generación autoregresiva: El código se genera token a token (palabra por palabra, símbolo por símbolo), donde cada token generado se añade al contexto para predecir el siguiente. Esto es exactamente el mismo mecanismo que usa GPT para generar texto.

  4. Fine-tuning con RLHF: El modelo fue refinado con feedback humano (Reinforcement Learning from Human Feedback), donde programadores evaluaron la calidad de las sugerencias, mejorando la relevancia y corrección del código generado.

Paso 3 — Normalización¶

¿Qué es? Convertir el texto a una forma estándar: minúsculas, sin caracteres especiales, sin números.

¿Por qué primero? La normalización debe hacerse antes de tokenizar para que 'INCREÍBLE' e 'increíble' sean tratadas como la misma palabra. Si tokenizamos primero, tendríamos tokens duplicados.

Decisión de diseño: Mantenemos los acentos (á, é, í, ó, ú) porque en español son semánticamente relevantes ('mas' ≠ 'más'). Eliminamos signos de puntuación especiales del español (¡¿) pero mantenemos la estructura de las palabras.

In [4]:
def normalizar(texto):
    """
    Normaliza un texto aplicando:
    1. Conversión a minúsculas
    2. Eliminación de signos de puntuación (incluidos ¡¿)
    3. Eliminación de números
    4. Eliminación de espacios múltiples
    """
    texto = texto.lower()                          # minúsculas
    texto = re.sub(r'[¡¿!?.,;:\"\'-]', ' ', texto) # eliminar puntuación
    texto = re.sub(r'\d+', '', texto)              # eliminar números
    texto = re.sub(r'\s+', ' ', texto).strip()     # espacios múltiples → uno
    return texto

Paso 5 — Eliminación de Stopwords¶

¿Qué son las stopwords? Palabras muy frecuentes que aportan poco significado semántico: 'de', 'la', 'que', 'y', 'a', 'en'...

¿Por qué eliminarlas? Reducen el ruido en modelos de clasificación y representación. Si no las eliminamos, 'de' y 'la' serían las palabras más frecuentes en cualquier corpus español, dominando las representaciones numéricas sin aportar información discriminativa.

Decisión de diseño: Usamos la lista de stopwords de NLTK para español. En un proyecto real, podríamos añadir stopwords específicas del dominio (ej: 'película', 'film' si solo analizamos reseñas de cine).

Paso 6 — Stemming¶

¿Qué es? Reducir cada palabra a su raíz morfológica mediante reglas heurísticas. No produce palabras reales, sino raíces.

Algoritmo: SnowballStemmer para español (también llamado Spanish Stemmer). Es una mejora del algoritmo de Porter, adaptado para lenguas romances.

Ventaja: Muy rápido, sin necesidad de diccionario. Desventaja: Puede producir raíces que no son palabras reales ('actuaron' → 'actu'), perdiendo legibilidad.

Paso 7 — Lematización¶

¿Qué es? Reducir cada palabra a su forma canónica (lema) usando un diccionario morfológico y análisis gramatical. A diferencia del stemming, el lema siempre es una palabra real.

Herramienta: spaCy con el modelo es_core_news_sm. Este modelo usa redes neuronales para determinar el lema correcto según el contexto gramatical.

Ejemplo clave: 'actuaron' → lema 'actuar' (correcto), mientras que el stemmer daría 'actu' (no es una palabra).

Nota importante: La lematización se aplica sobre el texto normalizado (no sobre los tokens ya filtrados) porque spaCy necesita el contexto de la oración para determinar el lema correcto.

def lematizar(texto):
    """
    Lematiza un texto usando spaCy.
    Retorna lista de lemas, excluyendo stopwords y puntuación.
    spaCy necesita el texto completo (no tokens) para analizar el contexto.
    """
    doc = nlp(texto)
    return [
        token.lemma_.lower()
        for token in doc
        if not token.is_stop       # excluir stopwords
        and not token.is_punct     # excluir puntuación
        and not token.is_space     # excluir espacios
        and len(token.text) > 1    # excluir tokens de 1 carácter
    ]

Análisis y conclusiones¶

Resultados observados¶

Normalización: La conversión a minúsculas y eliminación de puntuación redujo el vocabulario efectivo al unificar variantes como 'INCREÍBLE' e 'increíble'. Los signos ¡¿ del español son especialmente importantes de eliminar porque NLTK no siempre los maneja bien en tokenización.

Tokenización: word_tokenize de NLTK funcionó correctamente para el español, separando correctamente palabras y signos de puntuación. La tokenización por oraciones (sent_tokenize) fue menos precisa con oraciones cortas separadas por puntos seguidos de mayúsculas.

Stopwords: La eliminación redujo el corpus aproximadamente un 35-40% en número de tokens. Esto es significativo: casi la mitad de las palabras en español son funcionales (artículos, preposiciones, conjunciones) y no aportan información discriminativa para clasificación.

Stemming vs Lematización — diferencia clave:

  • El stemmer convirtió 'actuaron' → 'actu', 'extraordinaria' → 'extraordinari'. Estas raíces no son palabras reales pero son útiles para agrupar variantes morfológicas.
  • La lematización convirtió 'actuaron' → 'actuar', 'extraordinaria' → 'extraordinario'. Produce formas reales del diccionario, más interpretables.
  • ¿Cuándo usar cada uno? Stemming: cuando la velocidad es crítica y la interpretabilidad no importa (motores de búsqueda a gran escala). Lematización: cuando necesitamos resultados interpretables o trabajamos con modelos que usan vocabulario real (embeddings preentrenados).

Limitaciones encontradas¶

  1. Modelo spaCy pequeño: es_core_news_sm es el modelo más pequeño de spaCy para español. En algunos casos lematizó incorrectamente palabras poco comunes. El modelo es_core_news_lg daría mejores resultados a costa de más memoria.

  2. Stopwords genéricas: La lista de NLTK no incluye stopwords específicas del dominio de reseñas de cine. En producción, añadiríamos palabras como 'película', 'film', 'ver' que son muy frecuentes en este dominio pero poco discriminativas.

  3. Acentos: Decidimos mantener los acentos porque en español son semánticamente relevantes. Sin embargo, en textos de redes sociales donde los usuarios frecuentemente omiten acentos, habría que normalizar también este aspecto.

Conclusión general¶

El preprocesamiento no es un paso trivial: cada decisión (¿elimino stopwords? ¿uso stemming o lematización? ¿mantengo números?) afecta directamente la calidad de los modelos posteriores. La mejor práctica es experimentar con y sin cada técnica y medir el impacto en la tarea final. Representación 1 — Bag of Words (BoW)¶

¿Qué es? Representa cada documento como un vector de conteos de palabras. El vocabulario es el conjunto de todas las palabras únicas del corpus. Cada posición del vector corresponde a una palabra, y el valor es cuántas veces aparece en ese documento.

Ventaja: Simple, rápido, interpretable. Desventaja: Ignora el orden de las palabras y la frecuencia relativa entre documentos. 'buena' y 'mala' tienen el mismo peso si aparecen el mismo número de veces.

Representación 2 — One-Hot Encoding¶

¿Qué es? Variante de BoW donde solo se registra la presencia (1) o ausencia (0) de cada palabra, sin contar repeticiones.

¿Cuándo usarlo? Cuando la frecuencia de aparición no es relevante, solo si la palabra aparece o no. Útil para vocabularios pequeños o cuando queremos evitar que palabras muy repetidas dominen la representación.

Diferencia con BoW: Si 'película' aparece 3 veces en un documento, BoW pone 3, One-Hot pone 1.

Representación 3 — TF (Term Frequency)¶

¿Qué es? La frecuencia de un término normalizada por la longitud del documento. Evita que documentos más largos tengan automáticamente valores más altos.

Fórmula: TF(t, d) = (número de veces que t aparece en d) / (total de términos en d)

Diferencia con BoW: BoW da conteos absolutos; TF da frecuencias relativas (proporciones entre 0 y 1).

def calcular_tf(corpus_tokenizado):
    """
    Calcula TF para cada término en cada documento.
    TF(t,d) = count(t en d) / len(d)
    Retorna lista de diccionarios {término: tf_value}
    """
    tf_corpus = []
    for doc in corpus_tokenizado:
        tokens = doc.split()
        total = len(tokens)
        conteos = Counter(tokens)
        tf = {term: count / total for term, count in conteos.items()}
        tf_corpus.append(tf)
    return tf_corpus

Representación 4 — IDF (Inverse Document Frequency)¶

¿Qué es? Mide qué tan rara o informativa es una palabra en el corpus. Palabras que aparecen en muchos documentos tienen IDF bajo (son comunes, poco discriminativas). Palabras que aparecen en pocos documentos tienen IDF alto (son específicas, muy informativas).

Fórmula: IDF(t) = log(N / df(t)) donde N = número de documentos, df(t) = documentos que contienen t.

Intuición: Si 'película' aparece en los 5 documentos, IDF = log(5/5) = 0. Si 'magistral' aparece solo en 1, IDF = log(5/1) = 1.6. Las palabras raras son más informativas.

def calcular_idf(corpus_tokenizado):
    """
    Calcula IDF para cada término del vocabulario.
    IDF(t) = log(N / df(t))
    Usamos log natural. Añadimos +1 al denominador para evitar división por cero.
    """
    N = len(corpus_tokenizado)
    # Contar en cuántos documentos aparece cada término
    df = Counter()
    for doc in corpus_tokenizado:
        tokens_unicos = set(doc.split())
        df.update(tokens_unicos)

    idf = {term: math.log(N / (count + 1)) for term, count in df.items()}
    return idf

Representación 5 — TF-IDF¶

¿Qué es? Combina TF e IDF: pondera cada término por su frecuencia en el documento (TF) multiplicada por su rareza en el corpus (IDF). Resultado: términos frecuentes en un documento pero raros en el corpus reciben el mayor peso.

Fórmula: TF-IDF(t, d) = TF(t, d) × IDF(t)

¿Por qué es mejor que BoW? BoW da el mismo peso a 'película' (aparece en todos los docs) y 'magistral' (aparece en uno). TF-IDF penaliza 'película' y premia 'magistral', capturando mejor qué hace único a cada documento.

Usamos TfidfVectorizer de scikit-learn que implementa una versión optimizada con suavizado.

Representación 6 — N-gramas (Bigramas y Trigramas)¶

¿Qué son? Secuencias contiguas de N palabras. Un bigrama es una secuencia de 2 palabras, un trigrama de 3.

¿Por qué son útiles? Capturan contexto local que BoW ignora. 'no buena' y 'muy buena' tienen significados opuestos, pero BoW las trataría igual (mismas palabras, diferente orden). Los bigramas capturan esta diferencia.

Ejemplo: 'no recomendaría' es un bigrama que captura negación, algo que BoW pierde completamente.

Observaciones clave¶

TF-IDF vs BoW: La diferencia más notable fue que TF-IDF asignó peso 0 a términos que aparecen en todos los documentos (como 'película'), mientras que BoW les daba el mismo peso que a términos únicos. Esto hace que TF-IDF sea mucho más discriminativo para clasificación.

N-gramas: Con solo 5 documentos cortos, el número de trigramas únicos fue muy bajo (casi uno por documento), lo que ilustra el problema de la maldición de la dimensionalidad: con corpus pequeños, los n-gramas de orden alto son demasiado escasos para ser útiles estadísticamente.

Similitud coseno: Los documentos positivos (Doc1, Doc4) mostraron mayor similitud entre sí que con los negativos, validando que TF-IDF captura información semántica relevante incluso sin entender el significado de las palabras.

Limitaciones¶

Todos estos métodos son representaciones dispersas (sparse): la mayoría de los valores son 0. Con vocabularios grandes (miles de palabras), esto es computacionalmente ineficiente. Los embeddings densos (Word2Vec, BERT) del Tema 5 resuelven este problema al representar palabras en espacios de menor dimensión pero más ricos semánticamente.

Conclusión¶

Para la tarea de clasificación de reseñas, TF-IDF con unigramas y bigramas sería la representación más adecuada de las vistas en este tema: captura la rareza de los términos, normaliza por longitud del documento, y los bigramas permiten capturar negaciones ('no recomendaría'). Esta es la representación que usaremos en el Tema 4 con los modelos clásicos.

def entrenar_evaluar(nombre, modelo, X_train, y_train, X_test, y_test):
    """
    Entrena un modelo, mide tiempos y calcula métricas de evaluación.
    Retorna diccionario con todos los resultados.
    """
    # Entrenamiento
    t0 = time.time()
    modelo.fit(X_train, y_train)
    t_train = time.time() - t0

    # Predicción
    t0 = time.time()
    y_pred = modelo.predict(X_test)
    t_pred = time.time() - t0

    # Métricas (macro: trata todas las clases por igual)
    return {
        'Modelo': nombre,
        'Accuracy': accuracy_score(y_test, y_pred),
        'Precision': precision_score(y_test, y_pred, average='macro'),
        'Recall': recall_score(y_test, y_pred, average='macro'),
        'F1': f1_score(y_test, y_pred, average='macro'),
        'T_train(s)': round(t_train, 3),
        'T_pred(s)': round(t_pred, 3),
        'y_pred': y_pred
    }

Comparación y reflexión¶

Análisis de resultados¶

Los resultados obtenidos en este experimento muestran un orden de rendimiento que merece análisis cuidadoso:

Naive Bayes obtuvo el mejor F1 (0.9006) y también el menor tiempo de entrenamiento (0.005s). Este resultado, aunque sorprendente frente a la intuición general, es coherente con las características del problema: el dataset 20 Newsgroups tiene categorías temáticamente muy distintas (espacio, hockey, política, informática), con vocabularios poco solapados. En este escenario, la asunción de independencia condicional de Naive Bayes no penaliza tanto, y la alta dimensionalidad del espacio TF-IDF favorece su funcionamiento. Además, el parámetro alpha=0.1 (suavizado bajo) permite que el modelo sea más discriminativo.

Regresión Logística y SVM (LinearSVC) obtienen resultados muy similares y ligeramente inferiores a Naive Bayes en este experimento concreto. En la literatura general, estos modelos lineales suelen superar a Naive Bayes en datasets más grandes y con categorías más similares entre sí. La diferencia aquí es pequeña (~1–1.5 puntos de F1) y podría invertirse con ajuste de hiperparámetros.

KNN es el modelo con peor rendimiento y mayor tiempo de predicción. Esto se debe a que:

  1. En espacios de alta dimensión, la distancia entre puntos pierde significado (maldición de la dimensionalidad).
  2. La predicción requiere calcular distancias con todos los puntos de entrenamiento (O(n) por predicción).
  3. Con vectores TF-IDF muy dispersos, la similitud coseno entre documentos de categorías distintas puede ser artificialmente alta.

Decisiones tomadas y justificación¶

  • ¿Por qué 4 categorías y no 20?
    Con 20 categorías el tiempo de entrenamiento de KNN sería prohibitivo, y la visualización de la matriz de confusión sería ilegible. 4 categorías permiten un análisis más claro.

  • ¿Por qué remove=('headers','footers','quotes')?
    Sin esta opción, el clasificador aprendería a identificar categorías por los metadatos del email (el campo From o Newsgroups), no por el contenido real. Esto daría una accuracy artificialmente alta pero sin valor real.

  • ¿Por qué sublinear_tf=True?
    Reduce el impacto de términos que aparecen cientos de veces en un documento, que de otro modo dominarían el vector.

Conclusión¶

En este experimento, Naive Bayes obtuvo el mejor resultado con F1 = 0.9006, superando a SVM y Regresión Logística. Esto ilustra un punto importante: el rendimiento relativo de los modelos depende del dataset concreto. En categorías temáticamente muy distintas con vocabularios separados, Naive Bayes puede competir o superar a modelos más complejos.

Para producción en un problema general de clasificación de texto, seguiría recomendando Regresión Logística como primera opción porque, además de rendimiento competitivo, produce probabilidades calibradas (no solo clases). Esto permite establecer umbrales de confianza y es más interpretable (los coeficientes indican qué palabras son más predictivas de cada categoría).

def tokens_to_indices(tokens, vocab, max_len):
    indices = [vocab.get(token, 0) for token in tokens]
    if len(indices) > max_len:
        return indices[:max_len]
    return indices + [0] * (max_len - len(indices))

class TextDataset(Dataset):
    def __init__(self, token_lists, labels, vocab, max_len):
        self.token_lists = token_lists
        self.labels = labels
        self.vocab = vocab
        self.max_len = max_len

    def __len__(self):
        return len(self.labels)

    def __getitem__(self, idx):
        tokens = self.token_lists[idx]
        x = torch.tensor(tokens_to_indices(tokens, self.vocab, self.max_len), dtype=torch.long)
        y = torch.tensor(self.labels[idx], dtype=torch.long)
        return x, y

class RNNClassifier(nn.Module):
    def __init__(self, embedding_matrix, hidden_dim, output_dim, n_layers=1, dropout=0.3):
        super().__init__()
        num_embeddings, embedding_dim = embedding_matrix.shape
        self.embedding = nn.Embedding.from_pretrained(torch.tensor(embedding_matrix, dtype=torch.float32), freeze=False, padding_idx=0)
        self.rnn = nn.LSTM(embedding_dim, hidden_dim, num_layers=n_layers, batch_first=True, dropout=dropout, bidirectional=True)
        self.classifier = nn.Sequential(
            nn.Dropout(dropout),
            nn.Linear(hidden_dim * 2, output_dim)
        )

    def forward(self, x):
        embedded = self.embedding(x)
        _, (hidden, _) = self.rnn(embedded)
        hidden_cat = torch.cat((hidden[-2], hidden[-1]), dim=1)
        return self.classifier(hidden_cat)

BATCH_SIZE = 32

def train_model(model, train_dataset, val_dataset, epochs):
    device = torch.device('cuda' if torch.cuda.is_available() else 'cpu')
    model.to(device)
    optimizer = torch.optim.Adam(model.parameters(), lr=LEARNING_RATE)
    criterion = nn.CrossEntropyLoss()

    train_loader = DataLoader(train_dataset, batch_size=BATCH_SIZE, shuffle=True)
    val_loader = DataLoader(val_dataset, batch_size=BATCH_SIZE)

    for epoch in range(epochs):
        model.train()
        epoch_loss = 0
        for x, y in train_loader:
            x, y = x.to(device), y.to(device)
            optimizer.zero_grad()
            logits = model(x)
            loss = criterion(logits, y)
            loss.backward()
            optimizer.step()
            epoch_loss += loss.item()
        print(f'Epoch {epoch+1}/{epochs} — loss={epoch_loss/len(train_loader):.4f}')

    model.eval()
    y_true, y_pred = [], []
    with torch.no_grad():
        for x, y in val_loader:
            x = x.to(device)
            logits = model(x)
            preds = logits.argmax(dim=1).cpu().numpy()
            y_true.extend(y.numpy())
            y_pred.extend(preds)

    return y_true, y_pred

11. Discusión y conclusiones¶

  • El modelo RNN con Word2Vec muestra cómo los embeddings capturan semántica en un espacio vectorial entrenado sobre corpus específico.
  • El Transformer tiende a generalizar mejor porque ya incorpora un preentrenamiento masivo y un encoder contextual que considera todo el texto.
  • La RNN es más eficiente en memoria y más fácil de entrenar en conjuntos pequeños, pero suele ser menos precisa cuando el texto es complejo.
  • El Transformer es más robusto frente a ambigüedad y contexto, aunque requiere más recursos y tiempo de entrenamiento.

Lecciones clave¶

  1. Word Embeddings son una excelente forma de representar palabras en un espacio denso y útil para modelos secuenciales.
  2. RNN + embeddings es una técnica clásica válida en problemas de texto, especialmente cuando se dispone de pocos datos.
  3. Transformers fine-tunados son el enfoque moderno recomendado para tareas de clasificación de texto, porque aprovechan conocimiento previo y contexto completo.
  4. La elección entre ambos depende de la disponibilidad de recursos, el tamaño del dataset y la necesidad de obtener resultados rápidos vs. resultados más precisos.

3. Enfoque clásico: reglas, diccionarios y expresiones regulares¶

PERSONAS = ['Pedro Almodóvar', 'El presidente de México']
ORGANIZACIONES = ['IBM', 'Glovo', 'Telefónica', 'ONU', 'Sevilla FC', 'Real Madrid', 'Festival de Cannes']
LUGARES = ['Madrid', 'Barcelona', 'Nueva York', 'México', 'Ramón Sánchez-Pizjuán']

DATE_PATTERN = re.compile(r'\b(\d{1,2} de [a-zA-Z]+ de \d{4}|la semana pasada|próxima conferencia)\b', re.IGNORECASE)


def extract_entities_classic(text):
    entities = []
    for persona in PERSONAS:
        if persona in text:
            entities.append({'entity': 'PER', 'value': persona, 'method': 'diccionario'})
    for org in ORGANIZACIONES:
        if org in text:
            entities.append({'entity': 'ORG', 'value': org, 'method': 'diccionario'})
    for lugar in LUGARES:
        if lugar in text:
            entities.append({'entity': 'LOC', 'value': lugar, 'method': 'diccionario'})
    for match in DATE_PATTERN.finditer(text):
        entities.append({'entity': 'DATE', 'value': match.group(0), 'method': 'regex'})
    return entities

for i, texto in enumerate(texts, 1):
    print(f'\n--- Texto {i} ---')
    for ent in extract_entities_classic(texto):
        print(ent)

7. Discusión del enfoque Transformer¶

El modelo dslim/bert-base-NER fue entrenado en inglés (CoNLL-2003), lo que explica varios de los errores observados al aplicarlo a textos en español:

  • Fragmentación de tokens con acentos: Pedro Almodóvar se detecta como Pedro Almod porque el tokenizador BERT no reconoce bien los caracteres acentuados del español, produciendo subwords como ##óvar que no se agrupan correctamente.
  • Errores de etiquetado por idioma: En el texto 4, El se clasifica como ORG y ##ueva aparece como fragmento suelto. Esto ocurre porque el modelo no comprende la estructura sintáctica del español.
  • Confusión semántica: Ramón Sánchez-Pizjuán (un estadio, LOC) se etiqueta como PER porque el modelo asocia ese patrón de nombre propio con personas en inglés.
  • Entidades parciales: Glovo se detecta como G y Barcelona se clasifica como ORG en lugar de LOC, errores directamente relacionados con el desajuste lingüístico.

Conclusión: Un modelo NER entrenado en inglés aplicado a español produce resultados degradados. Para producción en español se debería usar un modelo específico como PlanTL-GOB-ES/roberta-base-cap3-ner o hacer fine-tuning sobre un corpus anotado en español.
